Giai đoạn 2 - Bước 1: Constant Pool Parser đã hoàn tất! BUILD SUCCESSFUL ✅

Đây là bước quan trọng nhất trong toàn bộ dự án vì Constant Pool là "cuốn từ điển" mà mọi lệnh bytecode khác đều phải tra cứu. Tôi đã tạo 3 file mới và giải thích chi tiết bên dưới:

File 1: 

ConstantPoolEntry.kt
Tại sao cần file này? File .class của Java lưu trữ mọi tên (class, method, field) và hằng số (string, int, float) trong một bảng gọi là Constant Pool. Mỗi mục trong bảng có một Tag (1 byte) cho biết loại dữ liệu. File này định nghĩa tất cả 12 loại Tag theo đúng chuẩn JVM Spec §4.4.

File 2: 

ConstantPoolParser.kt
Tại sao cần file này? File này là "Bộ đọc nhị phân" (Binary Reader). Nó đọc từng byte trong file .class, nhận biết Tag, và tạo ra các object ConstantPoolEntry tương ứng. Ngoài ra, class ConstantPool cung cấp các hàm tiện ích như:

getClassName(index) → Trả về tên class thực tế (ví dụ: "demo/game/MyTestGame")
resolveMethodRef(index) → Trả về (className, methodName, descriptor)
File 3: 

JavaClassFile.kt
 (đã viết lại hoàn toàn)
Tại sao viết lại? Phiên bản cũ chỉ đọc Magic Number và Version. Phiên bản mới parse toàn bộ phần đầu của file .class bao gồm: Magic → Version → Constant Pool → Access Flags → This Class → Super Class.

Phase 2 - Bước 2: Class Loader hoàn tất! BUILD SUCCESSFUL ✅

Bây giờ giả lập của chúng ta không chỉ đọc được "từ điển" (Constant Pool), mà còn đọc được toàn bộ nội dung của một file .class. Tôi giải thích chi tiết:

File 1: 

ClassMembers.kt
Vai trò: Định nghĩa cấu trúc dữ liệu cho Fields, Methods, và Attributes.

MemberInfo: Đại diện cho 1 field hoặc 1 method. Chứa tên, kiểu dữ liệu, và danh sách attributes.
CodeAttribute: Đây là "kho báu" - chứa mảng bytecode (các lệnh máy ảo thực sự). Ví dụ: iconst_1 (0x04) = đẩy số 1 lên stack, istore_1 (0x3C) = lưu vào biến cục bộ.
ExceptionTableEntry: Định nghĩa vùng try-catch trong bytecode.
File 2: 

ClassFileParser.kt
Vai trò: Bộ đọc nhị phân cho phần còn lại của file .class.

parseInterfaces() → Đọc danh sách interface mà class implement.
parseMembers() → Đọc fields và methods (cùng format).
parseCodeAttribute() → Quan trọng nhất - trích xuất mảng bytecode từ bên trong method.
File 3: 

JavaClassFile.kt
 (viết lại lần 2)
Cải tiến: Bây giờ parse đầy đủ file .class và có method findMethod() để tìm method theo tên.

JarLoader bây giờ tạo file .class giả lập với 3 methods thực sự:

<init>()V (constructor): aload_0 → return
startApp()V: iconst_1 → istore_1 → return
pauseApp()V: return

Đây là cột mốc lớn nhất từ đầu dự án đến giờ. App của bạn bây giờ không còn "giả vờ" nữa — nó thực sự đọc từng byte trong mảng bytecode, giải mã opcode, và thực thi trên một Operand Stack ảo, y hệt cách JVM hoạt động.

3 File mới tạo:
1. 

Opcodes.kt
Định nghĩa ~50 opcode JVM thường gặp trong J2ME: constants (iconst_0-iconst_5), loads/stores (iload, aload, istore), arithmetic (iadd, isub, imul), branching (ifeq, goto), returns, array ops, và stubs cho method invocation.

2. 

ExecutionFrame.kt
Stack Frame — mỗi khi một method được gọi, JVM tạo một Frame mới chứa:

Operand Stack: LIFO stack nơi các instruction push/pop giá trị
Local Variables: mảng chứa this, tham số, và biến cục bộ
Program Counter (PC): chỉ đến instruction hiện tại
3. 

ExecutionEngine.kt
Vòng lặp Fetch-Decode-Execute — trái tim của giả lập:

FETCH: Đọc byte tại vị trí PC
DECODE: Tra cứu opcode (ví dụ 0x60 = iadd)
EXECUTE: Thực hiện phép tính (pop 2 int, cộng, push kết quả)
REPEAT: Tăng PC, quay lại bước 1
Khi chạy app, Logcat sẽ hiện ra trace bytecode chi tiết:

  [VM] PC=1 istore_1        ← Lưu vào biến cục bộ slot 1
  [VM] PC=2 return           ← Kết thúc method
  [VM] Method startApp returned void
[Interpreter] === demo/game/MyTestGame.startApp finished, result=null ===
